## 需求分析

### 目标
在 Markdown 源码编辑器（`PaneSourceEditor.vue`）中，对 Markdown 语法符号进行可视化高亮，提升编辑体验。

### 需高亮的符号类型

| 类别 | 符号示例 | 说明 |
|------|----------|------|
| 强调 | `**`, `__`, `*`, `_`, `~~` | 粗体、斜体、删除线 |
| 行内代码 | `` ` ``, ``` `` ``` | 代码标记 |
| 链接/图片 | `[`, `]`, `(`, `)`, `!` | 链接和图片语法 |
| 标题 | `#`, `##`, etc. | 标题层级 |
| 引用 | `>` | 块引用 |
| 列表 | `-`, `*`, `+`, `1.` | 无序/有序列表 |
| 分隔线 | `---`, `***`, `___` | 水平线 |
| 转义 | `\` | 转义字符 |
| 表格 | `|`, `:` | 表格语法 |
| 任务列表 | `- [ ]`, `- [x]` | checkbox |

## 现有代码分析

### 高亮相关文件

| 文件 | 功能 |
|------|------|
| `adapters/sourceEditorCodeBlockHighlight.ts` | 代码块语法高亮（已有） |
| `adapters/sourceEditorSearch.ts` | 搜索结果高亮（已有） |
| `components/PaneSourceEditor.vue` | 编辑器主组件 |

### 现有实现模式

**`sourceEditorCodeBlockHighlight.ts`** 采用的模式：

1. 使用正则表达式解析内容（`getOpeningFence`, `isClosingFence`）
2. 调用 `lowlight` 对代码内容进行语法分析
3. 通过 `RangeSetBuilder` + `Decoration.mark` 构建装饰区间
4. 使用 `ViewPlugin` 在文档变化时更新装饰

```typescript
// 核心流程
getSourceCodeBlockHighlightRanges(content)  // 解析 → 范围数组
  → createHighlightDecoration(className)    // 创建 Decoration
  → ViewPlugin.decorations                   // 挂载到编辑器
```

### 当前 Markdown 扩展

在 `PaneSourceEditor.vue` 的 `createEditorExtensions()` 中：

```typescript
[
  history(),
  markdown(),                              // ← 仅解析结构，无高亮
  createSourceCodeBlockHighlightExtension(), // 代码块高亮
  createSourceEditorSearchExtension(),     // 搜索高亮
  keymap.of([...]),
  createSourceEditorLayoutTheme(),
  // ...
]
```

**问题**: `markdown()` 来自 `@codemirror/lang-markdown`，仅解析 AST，不输出高亮装饰。

## 实现方案

### 方案 A: Decoration + 正则匹配（推荐）

**思路**: 类似 `sourceEditorCodeBlockHighlight.ts`，用正则表达式匹配 Markdown 符号，生成装饰区间。

**优点**:
- 实现简单，易于理解和维护
- 与现有代码风格一致
- 可精确控制高亮范围和样式
- 性能可控（仅在高亮内容变化时更新）

**缺点**:
- 需要手动维护 Markdown 语法正则
- 可能与编辑器内置解析冲突（需测试）

**实现位置**: `adapters/sourceEditorMarkdownHighlight.ts`（新建）

**核心数据结构**:
```typescript
interface MarkdownHighlightRange {
  className: string;  // e.g., 'md-bold', 'md-link-bracket'
  from: number;
  to: number;
}
```

**高亮分类设计**:
| Class Name | 匹配符号 |
|------------|----------|
| `md-heading` | `^#{1,6}\s` |
| `md-bold` | `\*\*[^*]+\*\*` |
| `md-italic` | `(?<!\*)\*[^*]+\*(?!\*)` 或 `__[^_]+__` |
| `md-strikethrough` | `~~[^~]+~~` |
| `md-inline-code` | `` `[^`]+` `` |
| `md-code-fence` | ``` ^```\|^~~~ |
| `md-blockquote` | `^\s*>` |
| `md-list-marker` | `^[\s]*[-*+]` 或 `^\s*\d+\.` |
| `md-link-bracket` | `[\[\]]` |
| `md-link-paren` | `[()] ` |
| `md-image-prefix` | `!` |
| `md-hr` | `^[-*_]{3,}\s*$` |
| `md-table` | `^\s*\|` |
| `md-tasklist` | `- \[[ x]\]` |
| `md-escape` | `\\.` |

### 方案 B: @lezer/highlight + 自定义 Language

**思路**: 复用 `@lezer/highlight`（已安装）和 CodeMirror 的 `syntaxHighlighting` 系统。

**优点**:
- 与 CodeMirror 生态深度集成
- 可复用现有的 Markdown 语言定义

**缺点**:
- 需要研究 `@lezer/highlight` 的标签系统
- 复杂度较高
- 可能需要覆写 markdown language 数据

### 方案 C: 仅 CSS 样式覆盖

**思路**: 不生成新装饰，复用 `markdown()` 解析出的语法树，通过 CSS 选择器高亮。

**问题**: CodeMirror 的 markdown 解析不会自动添加高亮类，**此方案不可行**。

## 推荐方案 A 的实施计划

### 第一阶段：基础设施

1. **创建 `adapters/sourceEditorMarkdownHighlight.ts`**
   - 定义 Markdown 符号的正则匹配规则
   - 实现 `getMarkdownHighlightRanges(content: string): MarkdownHighlightRange[]`
   - 实现 `createMarkdownHighlightDecorations(content: string): DecorationSet`
   - 实现 `createSourceEditorMarkdownHighlightExtension(): Extension`

2. **定义 CSS 变量**（在现有主题中添加）
   ```css
   :root {
     --md-heading-color: #1a1a1a;
     --md-bold-color: #d73a49;
     --md-italic-color: #6f42c1;
     --md-code-color: #032f62;
     --md-link-color: #0060df;
     /* ... */
   }
   ```

3. **在 `PaneSourceEditor.vue` 中集成**
   ```typescript
   function createEditorExtensions(): Extension[] {
     return [
       // ...
       markdown(),
       createSourceEditorMarkdownHighlightExtension(),  // ← 新增
       createSourceCodeBlockHighlightExtension(),
       // ...
     ];
   }
   ```

4. **在组件样式中添加高亮类样式**
   ```less
   .md-heading { color: var(--md-heading-color); font-weight: bold; }
   .md-bold { color: var(--md-bold-color); font-weight: bold; }
   .md-italic { color: var(--md-italic-color); font-style: italic; }
   /* ... */
   ```

### 第二阶段：精细化

1. **优化正则，避免误匹配**
   - 行内代码内的符号不应高亮（如 `` `**bold**` `` 中的 `**`）
   - URL 中的特殊字符应排除

2. **处理嵌套情况**
   - `**_bold and italic_**` 应正确解析
   - 链接中的 Markdown 符号不应高亮

3. **性能优化**
   - 使用 `RangesetBuilder` 合并相邻同类高亮
   - 仅在相关行变化时重新计算（可选优化）

### 第三阶段：测试与调优

1. **测试各种 Markdown 场景**
   - 标题、列表、引用、代码块
   - 嵌套结构
   - 表格
   - 转义字符

2. **性能测试**
   - 大文档（>10万字符）的渲染性能

## 风险评估

| 风险 | 级别 | 应对 |
|------|------|------|
| 正则匹配与 CodeMirror 解析冲突 | 中 | 通过测试验证，必要时调整匹配策略 |

| 主题变量冲突 | 低 | 使用特定前缀 `md-` |
